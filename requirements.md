Спецификация: веб-панель поставщика (Supplier Portal)
1. Обзор проекта
Веб-приложение для поставщиков платформы Iceberg: вход по логину/паролю, управление каталогом товаров, отправка продуктов на модерацию, просмотр статусов модерации, замечаний администратора, отзывов и возвратов. Бэкенд уже реализован (Node.js + Prisma + PostgreSQL); панель — отдельный SPA, который работает с существующим API.

2. Анализ текущего серверного API
2.1 Роли и авторизация
Роль пользователя: SUPPLIER.
В auth.middleware.js при успешной проверке JWT в req.user подставляется пользователь с user.supplier (профиль поставщика).
Ожидаемый формат: Authorization: Bearer <accessToken>.
Refresh token: отдельный endpoint для обновления пары токенов (реализован в TokenService).
Для панели нужны:

Логин по email или номеру телефона + пароль (POST /api/auth/login: поле `email` принимает и email, и телефон — бэкенд определяет по формату) → выдача accessToken и refreshToken.
Вход по СМС-коду (без пароля): POST /api/auth/initiate-phone-login { phone } → loginToken; затем POST /api/auth/verify-phone-login { loginToken, verificationCode } → accessToken, refreshToken.
Сохранение токенов (например, в памяти + sessionStorage для refresh), автоматическое обновление по refreshToken при 401 (POST /api/auth/refresh-token).
Редирект на логин при неавторизованном доступе и при роли не SUPPLIER.
2.2 Продукты (модерация и каталог поставщика)
Используются маршруты из product.routes.js и логика из product.controller.js.

Метод	Endpoint	Описание
GET	/api/products	Список продуктов. Для SUPPLIER возвращаются только свои, включая неактивные и со статусом модерации. Пагинация: page, limit.
GET	/api/products/:id	Один продукт по ID (в т.ч. свои с moderationStatus PENDING/REJECTED).
POST	/api/products	Создание продукта. Для поставщика: supplierId не передаётся, продукт создаётся с moderationStatus: PENDING, isActive: false. Формат: multipart/form-data (поля + до 10 файлов images).
POST	/api/products/preupload-image	Предзагрузка одного изображения (для пошагового создания).
POST	/api/products/:productId/images	Добавление одного изображения к существующему продукту.
POST	/api/products/:productId/image-base64	Загрузка изображения в base64 (тело JSON).
POST	/api/products/:productId/finalize	Финализация создания продукта с заранее загруженными изображениями (тело: images — массив путей).
PUT	/api/products/:id	Обновление продукта. Для поставщика: нельзя менять supplierId; при любом изменении продукт снова уходит на модерацию (moderationStatus: PENDING, isActive: false).
DELETE	/api/products/:id	Удаление. Поставщик может удалять только свои продукты.
PATCH	/api/products/bulk/box-prices	Массовое обновление цен за коробку (до 100 позиций).
Поля продукта (схема Prisma и контроллер):

Обязательные: name, price, itemsPerBox, stockQuantity.
Опциональные: description, boxPrice, weight, categories (массив ID или строка "[1,2,3]" / "1,2,3"), warehouses (строка "all" или JSON массив объектов с warehouseId и опционально quantity, warehousePrice).
Для поставщика при создании/редактировании используются также supplierProposedPrice и supplierProposedBoxPrice; финальные price/boxPrice выставляет администратор при модерации.
Статусы модерации: PENDING, APPROVED, REJECTED. Есть поля moderationReason, moderatedAt, moderatedById.
Важно для панели:

После создания/редактирования показывать статус «На модерации» и при отклонении — текст из moderationReason.
Реализовать создание продукта в два шага: предзагрузка изображений и финализация — или одной формой с multipart/form-data.
Список категорий: в текущем коде отдельный GET-список категорий в приложении не найден. Либо добавляется endpoint типа GET /api/categories, либо категории запрашиваются из другого доступного API (если есть). В требованиях ниже заложен отдельный запрос списка категорий для выбора при создании/редактировании продукта.
2.3 Возвраты товаров (product returns)
Из productReturn.controller.js:

Метод	Endpoint	Описание
GET	/api/product-returns	История возвратов. Для SUPPLIER — только свои. Query: productId, warehouseId, status, page, limit.
GET	/api/product-returns/:id	Детали возврата (поставщик — только свои).
GET	/api/product-returns/stagnant	Застоявшиеся товары (дни без продаж). Query: daysThreshold, warehouseId, supplierId (для поставщика подставляется свой).
GET	/api/product-returns/statistics	Статистика по возвратам (поставщик — только свои).
PUT	/api/product-returns/:id/start	Перевести возврат в статус IN_PROGRESS (поставщик — только свои, только при статусе APPROVED).
Создавать запрос на возврат (POST) могут только ADMIN/EMPLOYEE; поставщик только просматривает и отмечает начало отгрузки.

2.4 Отзывы (feedback)
Из feedback.routes.js и контроллера:

Метод	Endpoint	Описание
GET	/api/feedbacks/supplier/:supplierId	Все отзывы по продуктам поставщика (для своей панели — передавать свой supplierId).
PUT	/api/feedbacks/:id/reply	Ответ поставщика на отзыв. Роль: только SUPPLIER.
Нужно: экран «Отзывы» (список + при необходимости фильтр по продукту) и форма ответа на отзыв.

2.5 Профиль поставщика
Из user.service.js и типичной структуры профиля:

Обновление профиля через общий endpoint профиля (например, PUT /api/profile или аналог), с проверкой роли на бэкенде. В User — email, avatar, gender; в Supplier — companyName, contactPerson, phone, address, inn, ogrn, bankAccount, bik.
Смена пароля — отдельный endpoint (например, PUT /api/profile/password или аналог), если он есть в вашем API.
В спецификации заложено: экран «Профиль» (просмотр и редактирование данных компании и контактов) и смена пароля.

2.6 Уведомления о модерации
В product.controller.js при создании/обновлении продукта поставщиком вызывается UniversalNotificationService.createModerationNotifications. Сами уведомления хранятся в модели Notification и привязаны к userId. Нужен endpoint вида GET /api/notifications (и опционально «прочитано») для отображения в панели. Если такого маршрута ещё нет — его нужно добавить в бэкенд и отразить в требованиях.

3. Функциональные требования к веб-панели
3.1 Аутентификация
Страница входа: два варианта — (1) email или номер телефона + пароль; (2) вход по коду из СМС (ввод телефона → запрос кода → ввод кода → выдача токенов).
После успешного входа — сохранение accessToken и refreshToken, редирект в кабинет.
При 401 — попытка обновить токен по refreshToken; при неудаче — выход и редирект на логин.
Выход (инвалидация токена на бэкенде, если есть такой endpoint, и очистка токенов на клиенте).
3.2 Кабинет (Dashboard)
Краткая сводка: количество товаров (всего / на модерации / одобренных / отклонённых), при желании — последние уведомления, последние отзывы.
Данные брать из GET /api/products (агрегация по moderationStatus) и при наличии — из уведомлений и отзывов.
3.3 Каталог продуктов
Список своих продуктов: таблица или карточки с пагинацией (GET /api/products?page=&limit=).
Фильтр по статусу модерации: PENDING, APPROVED, REJECTED.
Для отклонённых — отображение причины (moderationReason).
Действия: «Создать продукт», «Редактировать», «Удалить» (с подтверждением). Редактирование ведёт к повторной отправке на модерацию.
3.4 Создание / редактирование продукта
Поля: название, описание, цена за штуку, кол-во штук в коробке, цена за коробку (опционально), кол-во коробок, вес (опционально), категории (множественный выбор из списка, получаемого с API), склады (если API отдаёт список складов для поставщика — выбор; иначе по документации API).
Изображения: несколько файлов (до 10), с превью и возможностью удалить до отправки. Либо один шаг с multipart, либо сценарий preupload + finalize.
Валидация на клиенте в соответствии с правилами бэкенда (длина названия, числа > 0, boxPrice не ниже 50% от price×itemsPerBox и т.д.).
После успешного создания/обновления — сообщение «Отправлено на модерацию» и переход к списку или к карточке продукта.
3.5 Возвраты товаров
Список возвратов: GET /api/product-returns с пагинацией и фильтрами (статус, склад, продукт).
Просмотр одного возврата: GET /api/product-returns/:id.
Для возвратов со статусом APPROVED — кнопка «Начать возврат» (PUT /api/product-returns/:id/start).
Опционально: экран «Застоявшиеся товары» (GET /api/product-returns/stagnant) и статистика (GET /api/product-returns/statistics).
3.6 Отзывы
Список отзывов по своим продуктам: GET /api/feedbacks/supplier/:supplierId.
Ответ на отзыв: форма с текстом ответа, отправка PUT /api/feedbacks/:id/reply.
3.7 Профиль и безопасность
Просмотр и редактирование данных компании и контактов (поля модели Supplier + при необходимости User).
Смена пароля (текущий + новый), если API это поддерживает.
3.8 Уведомления
Блок или страница уведомлений (модерация, возможно другие типы), пометка прочитанных при наличии соответствующего API.
4. Нефункциональные требования
Адаптивная вёрстка: удобная работа с планшета и десктопа.
Локализация: русский язык интерфейса.
Обработка ошибок API: отображение сообщений от бэкенда (например, message в теле ответа), общий обработчик сетевых ошибок и 401/403.
Безопасность: не хранить пароль в хранилище; токены — в памяти + при необходимости в sessionStorage (или только в памяти для строгой безопасности).
5. Стек и технологии (рекомендации)
5.1 Базовый стек
React 18+ (функциональные компоненты, хуки).
TypeScript — для типизации моделей API (Product, Supplier, ModerationStatus, Feedback, ProductReturn и т.д.) и снижения ошибок.
5.2 Управление состоянием
Вместо Redux для такой панели достаточно:
React Query (TanStack Query) — кэш запросов к API, повторные запросы, инвалидация после мутаций (создание/редактирование продукта, ответ на отзыв, смена статуса возврата). Это покрывает большую часть глобального «состояния» (списки продуктов, возвратов, отзывов, профиль).
Локальный state (useState/useReducer) для форм и UI (модалки, шаги создания продукта).
Если позже появятся сложные общие сценарии (многошаговый визард с общим контекстом, сложный общий UI state), можно добавить Zustand (лёгкий store) или Jotai, не обязательно Redux.
Рекомендация: React Query + Zustand (минимально) или только React Query и контекст для темы/языка при необходимости.

5.3 Маршрутизация и запросы
React Router v6 — маршруты: /login, /, /products, /products/new, /products/:id/edit, /returns, /returns/:id, /feedbacks, /profile, при необходимости /notifications.
Fetch или axios — обёрнутые в клиент с подстановкой Authorization: Bearer ... и перехватом 401 с попыткой refresh.
5.4 UI-библиотека
Ant Design (antd) — много готовых компонентов (таблицы, формы, модалки, загрузка файлов, пагинация), нормальная поддержка TypeScript и темизации. Хорошо подходит для админских/поставщицких панелей.
Альтернативы: MUI (Material UI) или Chakra UI — более современный вид и гибкость; Mantine — удобные формы и даты. Выбор можно сделать по дизайн-гайдам вашего продукта.
Рекомендация: Ant Design или Mantine для быстрой и стабильной разработки панели.

5.5 Формы и валидация
React Hook Form + Zod (или Yup): типизированная валидация, соответствие правилам бэкенда (длина полей, числа, формат категорий и складов).
5.6 Сборка и окружение
Vite — сборка и dev-сервер.
Переменные окружения: VITE_API_BASE_URL, при необходимости отдельный базовый путь для панели поставщика, если он отличается от основного API.
6. Структура проекта (предложенная)
src/
  api/           # клиент HTTP, interceptors (token, refresh)
  components/    # переиспользуемые UI (таблицы, карточки продукта, загрузка изображений)
  features/      # модули: auth, products, returns, feedbacks, profile
  hooks/         # useAuth, useProducts, useProductReturns и т.д. (обёртки над React Query)
  pages/         # страницы маршрутов (Login, Dashboard, ProductList, ProductForm, Returns, Feedbacks, Profile)
  routes/        # конфигурация React Router, ProtectedRoute (проверка SUPPLIER)
  store/         # при использовании Zustand — слайсы (auth, ui)
  types/         # TypeScript-типы по API (Product, Supplier, ModerationStatus, Feedback, ProductReturn)
  utils/         # форматирование дат, чисел, константы статусов
7. Зависимости от бэкенда
Подтвердить или реализовать:
Endpoint входа (email + password) и выдачи пары access/refresh.
Endpoint обновления токена по refresh token.
Endpoint выхода (если предусмотрен).
Список категорий для выбора при создании/редактировании продукта: GET /api/categories (или эквивалент). Если его нет — добавить на бэкенде и описать в этой спецификации.
Список складов для выбора при создании продукта (если нужен в UI): проверить наличие GET складов с доступом для поставщика или общий endpoint.
Уведомления: GET /api/notifications (и PATCH/PUT для «прочитано»), при необходимости — добавить в бэкенд.
8. Этапы реализации (кратко)
Настройка проекта (Vite + React + TS), роутер, базовый layout (sidebar/menu, header с выходом).
Авторизация: страница логина, HTTP-клиент с токенами и refresh, защищённые маршруты.
Профиль: получение текущего пользователя/поставщика, форма редактирования, смена пароля.
Продукты: список с фильтром по статусу модерации, пагинация; создание и редактирование (форма + загрузка изображений); удаление.
Возвраты: список, детальная страница, кнопка «Начать возврат»; при необходимости — застоявшиеся товары и статистика.
Отзывы: список, ответ на отзыв.
Dashboard и уведомления (если API готов).
Тестирование и доработка ошибок/граничных случаев.
Итог: спецификация опирается на ваш текущий серверный код (контроллеры продуктов, возвратов, отзывов, схему Prisma и маршруты) и задаёт веб-панель на React с TypeScript, React Query (и при необходимости Zustand), современной UI-библиотекой (Ant Design или Mantine) и без обязательного использования Redux.
